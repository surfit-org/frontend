# 도메인 기반 아키텍처 가이드

우리 프로젝트는 기능(Function) 단위가 아닌 **도메인(Domain) 단위**로 코드를 구성합니다.

"내 코드를 어디에, 어떻게 작성해야 하는가?"

---

### 1. 왜 도메인 기반 구조인가요?

- **기능별 분류 (기본형)**: `components`, `hooks`, `services` 폴더에 모든 도메인 파일이 섞여 있어, 프로젝트가 커지면 탐색이 어렵습니다.
- **도메인별 분류 (우리 방식)**: `auth`, `surf` 등 비즈니스 단위로 폴더를 나눕니다. 관련 코드가 한 곳에 모여 **응집도가 높고 유지보수가 편합니다.**

---

### 2. 전체 폴더 구조

```
frontend/
├── app/                  # 화면 라우팅 (Expo Router)
│   ├── _layout.tsx       # 루트 레이아웃 (Provider 설정)
│   └── (tabs)/           # 하단 탭 네비게이션
│       └── index.tsx     # 홈 화면
│
├── features/             # 도메인별 기능 모음 ← 가장 많이 작업하는 곳
│   └── auth/             # 예시: 인증 도메인
│       ├── components/   # 해당 도메인 전용 UI 조각
│       ├── hooks/        # TanStack Query 훅
│       ├── services/     # API 호출 함수
│       └── types.ts      # 도메인 전용 타입
│
├── components/           # 여러 도메인에서 공통으로 쓰는 UI
│   └── ui/               # 버튼, 아이콘 등 기본 요소
│
├── stores/               # 전역 상태 (Zustand)
├── api/                  # openapi-fetch 클라이언트 설정
├── types/                # Swagger 자동 생성 타입 (수정 금지)
├── hooks/                # 공통 커스텀 훅
├── constants/            # 색상, 폰트 등 고정값
└── assets/               # 이미지, 아이콘 파일
```

---

### 3. 폴더별 역할

| 폴더                         | 역할                                         | 백엔드 대응 |
| :--------------------------- | :------------------------------------------- | :---------- |
| **app/**                     | 화면 라우팅. 파일명이 곧 URL 경로            | Controller  |
| **features/xxx/services/**   | API 호출 함수                                | Service     |
| **features/xxx/hooks/**      | 서버 데이터 페칭 및 캐싱 (TanStack Query)    | Repository  |
| **features/xxx/components/** | 해당 도메인 전용 UI 조각                     | DTO         |
| **stores/**                  | 앱 전체가 공유하는 클라이언트 상태 (토큰 등) | —           |
| **components/**              | 도메인과 무관한 공통 UI 컴포넌트             | global/util |

---

### 4. 코드 작성 규칙

#### 1: 화면은 `app/`에, 로직은 `features/`에

`app/` 파일은 얇게 유지합니다. 데이터 페칭과 비즈니스 로직은 `features/` 안으로 분리합니다.

```tsx
// ✅ app/(tabs)/surf.tsx — 얇게 유지
import { SurfScreen } from '@/features/surf/components/surf-screen';

export default SurfScreen;
```

#### 2: 공통 컴포넌트는 `components/`에

두 개 이상의 도메인에서 사용하는 UI 조각은 `features/` 밖으로 꺼냅니다.

```
components/
├── themed-text.tsx    # 다크모드 대응 텍스트
├── themed-view.tsx    # 다크모드 대응 뷰
└── ui/
    └── loading-view.tsx  # 공통 로딩 UI
```

#### 3: 전역 상태는 `stores/`에

화면 간 공유되는 데이터(로그인 토큰, 사용자 정보 등)는 Zustand store로 관리합니다.
화면 내부에서만 쓰이는 상태는 `useState`로 충분합니다.

---

### 5. 새 기능 추가 흐름 예시

`surf` 도메인(서핑 스팟 목록)을 새로 만드는 경우:

```
1. npm run gen:api               # 백엔드 API 타입 갱신

2. features/surf/ 폴더 생성
   ├── services/surf.service.ts  # getSurfSpots() API 호출 함수 작성
   ├── hooks/use-surf-spots.ts   # useQuery로 데이터 페칭 훅 작성
   └── components/surf-screen.tsx # UI 작성

3. app/(tabs)/surf.tsx 생성      # 라우팅 연결
```
