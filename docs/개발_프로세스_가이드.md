# 개발 프로세스

---

## **STEP 1. 이슈 생성**

기능 개발을 시작하기 전, 반드시 해당 작업에 대한 **Issue**를 먼저 만들기.

- **왜 하나요?**: 무엇을 만들지 팀원들에게 선언하고 기록을 남기기 위함.
- **약속**:
  - 이슈 템플릿에 맞춰 작성
  - 해당 작업에 맞는 **Label** 등록!!!
  - 이슈 생성 시 **Project** 보드에 등록되어 진행 상황 공유됩니다.

## **STEP 2. 브랜치 생성**

생성한 이슈 번호를 기반으로 브랜치 컨벤션에 맞춰 작업 공간을 만듭니다.

- **형식**: `feat/기능명` 또는 `fix/버그명`
- **예시**: `feat/login-api`
- develop 브랜치에서 push 안 되니 브랜치 만들어서 해야해요
- **Branch Protection**: `main`과 `develop` 브랜치에 대한 직접 푸시(Push)는 금지되어 있습니다. **PR**을 통해서만 합치기

## **STEP 3. 개발 및 커밋**

작업을 진행하며 의미 있는 단위로 커밋을 남깁니다.

- **약속**: 커밋 메시지 컨벤션을 준수합니다. (`feat: 로그인 API 구현`)
- 백엔드는 작업 시 **Swagger(@Tag, @Operation)**를 적극 활용해 문서화를 병행해주세용.

## **STEP 4. PR 생성 및 리뷰**

작업이 완료되면 `develop` 브랜치로 **Pull Request**를 보냅니다.

- **PR 템플릿**: 양식에 맞춰 작성
- **이슈 연결**: 본문에 `Closes #이슈번호`를 입력합니다. (Merge 시 해당 이슈 자동 종료)
- **리뷰**: 셀프 머지가 되긴 하는데, 팀원이 팔로업을 위해 PR을 확인하는 시간을 가집니다.
